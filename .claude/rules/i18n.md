# i18n — no hardcoded strings (ES/EN)

## Principle

**No user-facing text may be hardcoded in a component.** All UI copy comes from the project's translation/strings system. There is no exception for Spanish, even though ES is the default language.

This site is bilingual:

- **ES** is the default language and the **source** language (content is written in ES first), served with **no URL prefix**.
- **EN** is the secondary language, translated after ES is approved, served under the **`/en/`** prefix.

Two distinct kinds of text:

1. **UI copy** (nav labels, buttons, section titles, error/empty states) → comes from the **strings system** (e.g. `src/i18n/` dictionaries) and/or `siteSettings` from Sanity.
2. **Editorial content** (posts, product names, categories, brand story) → comes from **Sanity**, queried per locale. Never hardcoded either.

---

## Correct pattern

```astro
---
// .astro — resolve the locale and a t() helper from the strings system
import { useTranslations, getLocale } from '@/i18n'
const locale = getLocale(Astro.url)   // 'es' | 'en'
const t = useTranslations(locale)
---

<a href={t('nav.blog.href')}>{t('nav.blog.label')}</a>
<p>{t('blog.empty')}</p>
```

```tsx
// island .tsx — receive the locale/strings as props from the .astro that mounts it
function ProductFilters({ t }: { t: Translator }) {
  return <button>{t('filters.apply')}</button>
}
```

Editorial content is fetched per locale via the Sanity layer:

```ts
const post = await getPostBySlug(slug, locale)
```

---

## What NOT to do

```astro
<!-- ❌ Hardcoded Spanish string -->
<button>Aplicar</button>
<p>No hay publicaciones todavía</p>

<!-- ❌ defaultValue / inline fallback text as a "temporary fix" -->
t('filters.apply', { defaultValue: 'Aplicar' })

<!-- ❌ Template literal with hardcoded text -->
{`${count} productos`}

<!-- ❌ Concatenation with hardcoded text -->
{'Mostrando ' + total + ' resultados'}
```

## Locale maintenance rules

1. **Always update both locales at the same time.** When you add a key, add it to the ES dictionary **and** the EN dictionary (e.g. `src/i18n/es.ts` and `src/i18n/en.ts`, or both locale entries in `src/i18n/ui.ts`). Never add a key in only one locale.
2. **ES is the source of truth for meaning**; EN is its translation. If you don't have the final EN copy yet, add a clearly-marked placeholder and flag it — do **not** leave the key missing and do **not** fall back to ES silently.
3. **Group keys by area** using nested keys (`nav.*`, `blog.*`, `product.*`, `brand.*`, `common.*`). Follow the existing dictionary hierarchy — don't introduce flat keys if the area already uses nesting.
4. **Every ES page ships an EN pair** under `src/pages/en/` with correct **hreflang** linking the two (see `PROJECT.md` route map and the SEO stage).

```ts
// ✅ Correct — nested key present in both locales
// src/i18n/es.ts
export default { blog: { empty: 'No hay publicaciones todavía' } }
// src/i18n/en.ts
export default { blog: { empty: 'No posts yet' } }

// ❌ Incorrect — flat key that breaks the hierarchy
{ blogEmpty: '...' }
```

---

## Allowed exceptions

The only Spanish strings outside the locale dictionaries / Sanity are:

- `console.log` / `console.error` messages aimed at the developer (never the user).
- Code comments.
- Mock/seed values in development-only files.

> The strings system's exact shape (helper names, file layout) is defined in the Core Layout & i18n stage. Whatever the final API, these rules hold: no hardcoded user-facing copy, both locales updated together, editorial content from Sanity, ES↔EN page pairs with hreflang.
