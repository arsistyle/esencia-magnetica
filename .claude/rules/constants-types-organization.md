# TypeScript constants and types organization

## Principle

Constants (non-markup values declared outside a component) and TypeScript types that do not belong to a single component **are never left inline in page or component files**. They are extracted to a `constants.ts` or a types file as appropriate.

Unlike utilities (where each function gets its own file), constants and types are consolidated into **one file per domain** — not one file per constant.

---

## Where each constant / type lives

| Scope                                          | Destination                                              |
| ---------------------------------------------- | -------------------------------------------------------- |
| Used in 2+ domains or cross-cutting by nature  | `src/lib/constants.ts` / `src/types/index.ts`            |
| Only makes sense within one domain             | `src/lib/<domain>/constants.ts` / `src/types/<domain>.ts` |

The promotion rule is the same as in `utils-organization`: born in the domain; if needed in a second domain, promoted to `src/lib/constants.ts` or `src/types/`.

---

## What to extract

### Constants (`constants.ts`)

- Option arrays (structural seeds like `NAV_ITEMS`, `SOCIAL_LINKS`).
- Config maps (`LOCALES`, route config).
- Named scalar values (`POSTS_PER_PAGE`, `DEFAULT_OG_IMAGE`).
- Any `UPPER_SNAKE_CASE` value declared outside a component.

> Remember: categories, tags, and product data are **managed in Sanity**, not hardcoded. Only structural site values (route config, locales, pagination limits) belong in `constants.ts` — never editorial content.

### Types (`src/types/`)

- Domain interfaces and types **not** already generated by Sanity (`Locale`, `NavItem`, `SeoMeta`).
- Component props reused across files.
- Domain union types.

**Do not extract** types that are purely local to a component and not used in any other file (inline props of an internal handler, for example).

> **Sanity types:** types for content documents (post, product, category, `siteSettings`) must come from **Sanity TypeGen**, not be written by hand. Import them from wherever they are generated (e.g. `src/sanity/types` or `@/lib/sanity`); do not duplicate them in `src/types/`.

---

## Expected structure

```
src/
├── types/
│   ├── index.ts            # types shared across domains (Locale, SeoMeta…)
│   └── <domain>.ts         # types for a specific domain (if applicable)
└── lib/
    ├── constants.ts        # cross-cutting site constants
    └── <domain>/
        └── constants.ts    # constants for a specific domain
```

---

## What NOT to do

```ts
// ❌ Domain constant defined inline in a component / page
// src/pages/productos.astro
const PRODUCT_CATEGORIES = ['Outfits', 'Tops', 'Pantalones', ...]  // → should come from Sanity

// ❌ Domain type written by hand that Sanity already generates
interface Post { _id: string; title: string; /* ... */ }

// ❌ One file per constant
// src/lib/blog/postsPerPage.ts  → NO
```

## What TO do

```ts
// ✅ Structural constant in src/lib/constants.ts
export const LOCALES = ['es', 'en'] as const
export const POSTS_PER_PAGE = 12

// ✅ Cross-cutting type in src/types/index.ts
export type Locale = (typeof LOCALES)[number]
export interface SeoMeta {
  title: string
  description: string
  ogImage?: string
}

// ✅ Direct import, no barrel
import { LOCALES, POSTS_PER_PAGE } from '@/lib/constants'
import type { Locale, SeoMeta } from '@/types'
```

---

## Relationship with `utils-organization`

| What it is                       | Where it goes                          |
| -------------------------------- | -------------------------------------- |
| Pure function                    | `src/lib/<name>.ts` (its own file)     |
| Constant / config value          | the domain's `constants.ts`            |
| Type / TS interface              | `src/types/` (or a Sanity-generated type) |

Functions follow the one-file-per-domain rule. Constants and types are consolidated into a single file per domain.
